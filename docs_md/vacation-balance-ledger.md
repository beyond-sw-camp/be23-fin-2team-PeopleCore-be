# 휴가 잔여 정합성 — 상태별 분리 + Ledger 원장 추적

> 휴가 잔여를 "적립 - 사용" 단일 모델이 아니라 **결재 상태별로 분리 보관**하고, 모든 변동을 **append-only Ledger 원장**으로 기록해 결재 흐름 어느 단계에서도 가용 잔여가 정확히 표현되도록 한 도메인 설계 문서입니다.

## 1. 개요

| 항목 | 내용 |
|------|------|
| 잔여 모델 | **상태별 분리** — 적립 / 사용 / 대기 / 만료 |
| 가용 잔여 정의 | `가용 = 적립 - (사용 + 대기 + 만료)` |
| 변동 기록 | **Ledger append-only 원장** — 변동 전후값 + 사유 + 참조 + 처리자 |
| 무결성 가드 | `@PrePersist` 부호 검증 (증가 이벤트 ≥ 0, 차감 이벤트 ≤ 0) |
| 결재 흐름 매핑 | 상태 전이별로 변동 메서드 책임 분리 |

---

## 2. 왜 단일 값 모델로 안 되는가

### 문제 사항
- 잔여를 "적립 - 사용" 단일 모델로 짜면 **결재 상태 전이 중간**에 잔여가 일시적으로 잘못 계산 → 사용자에게 잘못된 가용 잔여 노출, UX 악화
- **만료된 잔여와 실제 사용된 잔여**가 한 값에 뭉쳐 있으면 미사용 수당 계산 시 만료분까지 보상되어 **회계 오류**
- 변동 이력 부재 시 정합성이 깨졌을 때 디버깅·감사·복구 수단 전무

### 원인 분석
- 잔여는 다양한 결재 상태 트리거에 의해 변경 → 단일 값은 어떤 트리거가 어떤 변동을 일으켰는지 표현 불가
- 단일 row mutation 만으로는 변동 시점·사유·전후값을 사후 검증할 수단이 없음 → 사고 발생 시 원인 추적 비용 누적

## 4. 해결 설계

### 4.1 상태별 분리 모델

```
가용 잔여 = 적립 - (사용 + 대기 + 만료)
```

| 컬럼 | 의미 | 결재 흐름 매핑 |
|------|------|----------------|
| 적립 | 부여된 총량 | 정책 부여 / 이월 적립 |
| 사용 | 결재 최종 승인 후 차감 | 휴가 결재 승인 완료 |
| 대기 | 결재 진행 중 임시 차감 | 휴가 결재 기안 → 승인 전 |
| 만료 | 유효기간 경과 분 | `BalanceExpiry` 배치 |

**핵심**: 결재 상태 전이 어느 단계에서도 가용 잔여가 의미 일관 유지 → 사용자에게 보이는 값이 항상 정확

### 4.2 결재 상태 전이별 변동 메서드 책임 분리

- 결재 흐름과 코드가 **일대일 대응**
- 각 메서드는 자기 책임의 컬럼만 변경 + Ledger 한 row 작성

| 결재 이벤트 | 잔여 변동 | Ledger 사유 |
|-------------|-----------|--------------|
| 휴가 신청 (기안) | 대기 ↑ | 결재 대기 임시 차감 |
| 결재 승인 | 대기 ↓, 사용 ↑ | 결재 최종 승인 |
| 결재 반려 / 회수 | 대기 ↓ | 결재 반려/회수 복원 |
| 정책 부여 | 적립 ↑ | 연간 부여 / 이월 |
| 유효기간 경과 | 만료 ↑ | 배치 만료 처리 |

### 4.3 Ledger append-only 원장

| 필드 | 역할 |
|------|------|
| 변동 종류 | 적립 / 사용 / 대기 / 만료 중 어느 컬럼인지 |
| 변동량 | 부호 포함 (+/-) |
| 변동 전/후 값 | 사후 검증 가능 |
| 사유 코드 | "결재 승인", "이월", "만료" 등 |
| 참조 | 결재 문서 ID 등 |
| 처리자 | 시스템 / 사원 / 관리자 |
| 처리 시각 | timestamp |

- **append-only** → 과거 row 수정/삭제 금지
- 변동 누적합 = 현재 잔여 → 정합성 사후 검증 가능
- 감사·복구·디버깅 모두 한 테이블에서

### 4.4 무결성 가드 — `@PrePersist` 부호 검증

```java
@PrePersist
private void validateSign() {
    // 증가 이벤트(적립 등)는 ≥ 0
    // 차감 이벤트(사용/대기/만료)는 ≤ 0
    // 팩토리 우회 직접 빌드 시에도 비정상 row 진입 차단
}
```

- 팩토리 메서드를 우회한 직접 빌드 시에도 비정상 row 진입 차단
- 영속화 직전 단계에서 최종 방어

## 5. 효과

| 지표 | Before (단일 값) | After (상태별 + Ledger) |
|------|------------------|--------------------------|
| 결재 대기 중 가용 잔여 노출 | 일시적 오류 | 정확 |
| 미사용 수당 계산 | 만료분 혼입 가능 | 사용분만 정확 보상 |
| 정합성 깨졌을 때 추적 | 수단 없음 | Ledger 로 변동 전후 추적 |
| 신규 변동 유형 추가 | 단일 컬럼 의미 오버로드 | 컬럼/메서드 추가만으로 확장 |
| 비정상 row 방어 | 호출부 의존 | `@PrePersist` 최종 가드 |

## 6. 운영 규칙

| 규칙 | 이유 |
|------|------|
| 잔여 변경은 **반드시** 책임 메서드 경유 (Ledger 동시 작성) | 단일 진입점 보장 |
| Ledger 는 append-only — 과거 row 수정/삭제 금지 | 감사 추적성 |
| 결재 상태 전이 시 대기 ↔ 사용 이동을 한 트랜잭션 내에서 | 일시적 음수 방지 |
| 신규 변동 유형은 부호 규약 명시 후 추가 | `@PrePersist` 가드 정합 |
